# tokenhabit — 습관 카탈로그 (25패턴)

> 8카테고리 25패턴 (R1 14 + R2 13에서 3쌍 완전중복 통합 → 24, v1.1에서 거대 디렉토리 자동 탐색 차단(H3-04) 추가 → 25). 무의식 시나리오 → 왜 새는가 → 자각신호 → 고치는습관 → 절감추정.
> R1(리서처1, 수동 패턴) / R2(리서처2, 능동 패턴) 표시.

---

## H1 — 세션 위생 (Session Hygiene)

---

### H1-01 · 주제 드래그 [R1 P-01]
작업 전환 없이 전혀 다른 주제를 같은 세션에서 이어가는 습관.

**왜 새는가** Claude Code는 매 턴 전체 히스토리를 API에 재전송한다. 이전 작업의 파일 읽기 결과·에러 스택·대화 내용이 새 작업의 매 턴 input 토큰으로 과금된다.

**자각신호** `/usage`에서 Messages 비율 40%+, Claude가 무관 파일을 "이전에 읽었다"며 언급, 세션 60분 초과+복수 주제.

**고치는습관** 작업 전환 전 항상 `/clear`. `/rename`으로 세션 이름 붙인 뒤 `/clear` → 나중에 `/resume` 복귀.

**절감추정** 50K 이전 컨텍스트 × 후속 턴수만큼 절감. 정성적 2~5× 비용 절감.

---

### H1-02 · 실패 컨텍스트 적체 [R1 P-02 / R2 B3]
오류 루프를 길게 돌리면서 `/compact`·`/clear`·`/rewind`를 쓰지 않는 습관.

**왜 새는가** 실패한 시도(코드 조각·에러 텍스트·재시도 설명)가 히스토리에 쌓이고 매 턴 재전송된다. "lost in the middle" 효과로 정확도도 떨어진다.

**자각신호** 같은 이슈 3턴 이상 보정 중, Claude가 이미 실패한 방법을 다시 제안, `/usage` 토큰 빠르게 증가.

**고치는습관** 2회 연속 보정 실패 시 즉시 `/compact "실패한 접근법 제외, 현재 코드 상태와 문제만 보존"`. 또는 `/rewind`(Esc+Esc)로 복원 — `/rewind`는 단순 롤백이 아니라 대화만/코드만/둘다 복원 + "Summarize from/up to here"식 부분 요약 옵션을 제공하므로, 전체 `/compact` 대신 실패 지점 기준으로 부분 압축이 가능하다. 복원 후 더 구체적 프롬프트로 재시작.

**절감추정** 실패 루프 5턴 × 20K 누적 = 100K 추가 input 토큰. `/compact` 타이밍에 따라 40~70% 절감.

---

### H1-03 · compaction 버스 막차 타기 [R1 P-03]
자동 compaction이 뜰 때까지 기다리다가 풀 컨텍스트로 수십 턴을 보내는 습관.

**왜 새는가** 자동 compaction은 컨텍스트가 상한에 가까워질 때 발동한다(정확 임계치는 공식 미공개). 발동 전까지는 전체 히스토리가 매 턴 input으로 과금되므로, 컨텍스트가 클수록 모든 턴이 비례적으로 비싸진다.

**자각신호** 상태바 컨텍스트 사용률이 눈에 띄게 높음, 세션이 길고 복잡 주제 지속, 응답 속도 저하+초반 지시 "잊는" 현상. 정량 기준: Chroma 2025 연구(18개 프론티어 모델 공통, 토큰 길이 대상)에 따르면 누적 약 50K 토큰부터 성능 저하(context rot)가 시작되며 200K 윈도우 절반 미만에서도 발생한다. (시간 임계는 이 연구의 결과가 아니다 — 별도 휴리스틱으로, 세션이 길수록 토큰이 누적되므로 30~40분 이상 이어진 세션은 한 번 점검할 것을 경험적으로 권장.) 상한을 기다리지 말고 50K 토큰 도달 시 선제 `/compact`.

**고치는습관** 세션이 길어지고 컨텍스트가 무겁다고 느껴지면 자동 발동을 기다리지 말고 `/compact [포커스 지시]` 수동 실행. CLAUDE.md에 "When compacting, preserve full list of modified files and test commands" 추가.

**절감추정** 컨텍스트가 클수록 발동 전까지 매 턴 비용이 비례 증가. 일찍 수동 compact하면 이후 턴의 input 비용을 크게 줄일 수 있다(정성적).

---

### H1-04 · 주방 싱크 세션 [R2 D2]
무관한 작업 여러 개를 한 세션에 혼재시키는 습관.

**왜 새는가** 작업 A의 파일들이 작업 B를 처리할 때도 컨텍스트 공간을 점유. Claude가 무관 히스토리를 참조하며 혼선. auto-compact 시 중요 정보 손실 위험.

**자각신호** 한 세션 내 토픽 2회 이상 전환, "다시 처음 얘기로 돌아가서...", `/context`에 여러 작업 파일 혼재.

**고치는습관** 작업 전환 시 `/clear`. `/rename`으로 세션 이름 지정(`oauth-bug`, `new-feature-spec`) → `/resume`으로 복귀.

**절감추정** 무관 컨텍스트 제거로 각 작업당 처리 토큰 20~40% 절감.

## H2 — 반복 입력 낭비 (Redundant Input)

---

### H2-01 · 파일 리드 재탕 [R1 P-04]
한 세션에서 같은 파일을 여러 번 Read 요청하는 습관.

**왜 새는가** Read 툴은 파일 전체를 툴 결과로 컨텍스트에 추가한다. 200줄 파일(≈2,000 토큰)을 5번 읽으면 동일 내용이 10,000 토큰으로 부풀어 매 후속 턴에 재전송된다.

**자각신호** 툴 호출 로그에 같은 파일명 반복, Read 결과가 대화에 여러 번 삽입, `/usage`에서 툴 결과 비율이 메시지 비율을 압도.

**고치는습관** "아까 읽은 auth.ts에서 토큰 리프레시 부분"처럼 파일 재읽기 없이 컨텍스트 내 참조 유도. PreToolUse 훅으로 이미 읽은 파일 재읽기 차단(실측 80~95% 절감 보고).

**절감추정** 200줄 파일 5회 재읽기 ≈ 8,000 토큰 절약. 대형 파일(500줄+) 반복 시 세션당 수만 토큰.

---

### H2-02 · 로그 전체 덤프 [R1 P-05]
빌드·테스트 출력 전체를 Ctrl+A → Ctrl+V로 프롬프트에 붙여넣는 습관.

**왜 새는가** 붙여넣기한 텍스트는 사용자 메시지로 히스토리에 영구 추가된다. 10,000줄 로그 ≈ 수만 토큰이 이후 모든 턴의 input에 포함된다.

**자각신호** 프롬프트에 수천 줄 텍스트를 그대로 붙임, Bash 툴 출력이 필터링 없이 노출, 다음 턴부터 응답이 현저히 느려짐.

**고치는습관** 에러 메시지 핵심 20~30줄만 선별해 붙이기. PreToolUse 훅으로 `grep -A5 'FAIL\|ERROR'`만 반환하도록 필터링(공식 docs 예시). 파일로 저장 후 경로만 전달: "에러 로그 `/tmp/build.log`에 저장했어."

**절감추정** 10,000줄 로그 → 30줄로 필터 = 99%+ 절감. 이후 10턴 재과금 방지까지 수십만 토큰 차이.

---

### H2-03 · 스크린샷 중독 [R1 P-06]
UI 버그 설명 때마다 전체 화면 스크린샷을 반복 첨부하는 습관.

**왜 새는가** 1,000×1,000px 스크린샷 ≈ 1,334 토큰. 이미지는 히스토리에 남아 이후 턴에 재전송된다. 텍스트로 충분히 표현 가능한 정보에 수천 토큰을 쓰는 셈.

**자각신호** 하나의 이슈 해결에 3개+ 이미지 첨부, 이미지 내 관심 영역이 전체의 10% 미만, 에러 메시지가 텍스트인데 스크린샷으로 첨부.

**고치는습관** 텍스트 대체 가능하면 텍스트로. 이미지 필수 시 크롭 후 첨부(200×200px ≈ 54 토큰, 1,000×1,000 대비 25배 절감). 동일 스크린샷 재첨부 금지 — "아까 첨부한 스크린샷 기준으로..."로 참조.

**절감추정** 1,000×1,000px × 5회 ≈ 6,670 토큰. 크롭+텍스트 대체 시 90%+ 절감.

---

## H3 — 상시 주입 컨텍스트 (Always-On Context)

---

### H3-01 · 과비대 CLAUDE.md [R1 P-07]
프로젝트 규칙·아키텍처·API 문서·도메인 지식을 CLAUDE.md 하나에 몽땅 몰아넣는 습관.

**왜 새는가** CLAUDE.md는 세션 시작 시 자동 로드되어 매 턴 input의 고정 비용이 된다. 불필요한 내용이 길어질수록 Claude가 실제로 필요한 지시를 "잃어버리는"(lost in the middle) 현상 심화. 공식 권고: "Keep it under 200 lines."

**자각신호** CLAUDE.md 200줄 초과, Claude가 명시한 규칙을 무시하는 빈도 증가, `/usage`에서 Memory files 비율 3% 초과, 현재 작업과 무관한 도메인 섹션 항상 포함.

**고치는습관** CLAUDE.md는 200줄·2,000 토큰 미만 유지. 특정 작업에만 필요한 지식은 Skill로 분리(`.claude/skills/`) — 온디맨드 로드, 미사용 시 이름/설명만 ≈200 토큰. `@path` 임포트는 필요 시점에만 명시적으로.

**절감추정** 4,000 토큰짜리 CLAUDE.md → 1,000 토큰 감량 = 100턴 세션이면 30만 토큰 절감.

---

### H3-02 · 안 쓰는 MCP 서버 상시 연결 [R1 P-08]
"혹시 필요할지 몰라서" 여러 MCP 서버를 한꺼번에 켜놓는 습관.

**왜 새는가** 2026 공식 문서 기준 MCP tool 정의는 **기본 deferred(지연 로드)** — Claude가 특정 tool을 실제로 호출하기 전까지는 tool 이름만 컨텍스트에 들어가고 전체 schema는 주입되지 않는다. 따라서 "모든 schema가 매 요청 상시 재전송"은 더 이상 사실이 아니다. 다만 다수 MCP 서버를 연결하면 tool 이름 목록·연결 초기화·실제 호출 시점의 schema 주입 비용이 누적되고, 안 쓰는 서버일수록 그 비용이 순수 낭비다. (일부 서버는 `alwaysLoad` 설정 등으로 deferred 예외가 되어 schema가 상시 주입될 수 있으니 확인하라.) (과거 non-deferred 환경에서는 Playwright ≈3,442 토큰, Gmail ≈2,640 토큰처럼 schema가 상시 주입됐다는 보고가 있으나, deferred 모드에서는 적용되지 않는다.)

**자각신호** `/context`에서 System tools 비율이 높음, 세션에서 실제로 쓰지 않는 MCP 서버가 절반 이상, 단순 코딩 세션인데도 시작이 느림.

**고치는습관** `/mcp` 명령으로 현재 연결 서버 확인 후 미사용 서버 비활성화 — 호출·초기화 비용과 tool 이름 목록 오버헤드를 줄인다. CLI 대체 가능한 경우 MCP보다 CLI 툴 우선(`gh`, `aws`).

**절감추정** deferred 모드에서는 "상시 N토큰 절감"이 아니라, 미사용 서버 정리로 호출·초기화 비용과 tool 이름 목록 부담을 줄이는 효과. 정량치는 환경(deferred 여부)·서버 수에 따라 달라진다.

---

### H3-03 · 대형 참조파일 매번 전체 주입 [R1 P-11]
CLAUDE.md에 `@docs/full-api-spec.md` 같은 대형 파일을 임포트해 놓는 습관.

**왜 새는가** CLAUDE.md의 `@path` 임포트는 조건부 로드가 아닌 자동 전체 로드다. 필요한 건 한 섹션뿐인데 전체 명세(5,000+ 토큰)가 매 턴 상주한다.

**자각신호** CLAUDE.md에 `@` 임포트 항목 3개 이상, 임포트된 파일 중 현재 작업과 무관한 것이 있음, `/context`에서 Memory files 비율이 높음.

**고치는습관** CLAUDE.md에서 `@` 임포트 제거. 필요 시 프롬프트에서 직접 `@docs/api-spec.md` 참조. 대형 레퍼런스는 Skill로 전환 — 온디맨드 호출.

**절감추정** 5,000 토큰짜리 참조파일 2개 제거 = 세션당 10,000 토큰 고정 절감. 매 세션 복리 효과.

---

### H3-04 · 거대 디렉토리 자동 탐색 미차단 [practices_scan 섹션1-A]
빌드 산출물·의존성 디렉토리를 Claude 자동 탐색에서 차단하지 않는 습관.

**왜 새는가** Claude는 작업 중 프로젝트 파일을 자동으로 탐색·읽는다. `node_modules/`, `.next/`, `dist/`, 빌드 산출물·lock 파일이 탐색 대상에 들어가면 무관한 대형 디렉토리가 컨텍스트로 빨려 들어가 부풀린다.

**자각신호** Claude가 `node_modules/`·`.next/`·`dist/` 내부 파일을 읽거나 그쪽을 grep함, 빌드 산출물 파일명이 탐색 로그에 등장.

**고치는습관**
- **(공식 답변) `settings.json`의 `permissions.deny`로 차단** — 거대 빌드/벤더 디렉토리를 deny 규칙으로 막는다. 예: `"permissions": { "deny": ["Read(./node_modules/**)", "Read(./.next/**)", "Read(./dist/**)"] }`. Anthropic이 디렉토리 제외 요청에 권한 거부를 공식 안내한다.
- **`.gitignore` 존중** — Claude Code는 일부 동작에서 `.gitignore`를 참고하는 것으로 알려져 있다(존중 범위는 동작·버전별로 다를 수 있으므로 이것에만 의존하지 말 것).
- (참고) `.claudeignore` 전용 파일은 커뮤니티가 요청 중이나(예: GitHub 이슈 다수) 현재까지 공식 미구현이다 — 쓰라고 권하지 않는다. 차단은 위의 `permissions.deny`로 한다.

**절감추정** 정성적(프로젝트별 상이). 빌드 산출물·벤더 디렉토리가 큰 프로젝트(특히 Next.js)일수록 자동 탐색 컨텍스트 부풀림을 크게 줄인다. 1회 설정으로 매 세션 복리 효과.

## H4 — Thinking 토큰 과다 사용 (Thinking Token Overuse)

---

### H4-01 · 추론의 망령 [R1 P-10]
복잡한 작업에서 thinking을 높게 쓴 뒤, effort를 낮추지 않고 단순 작업도 같은 설정으로 이어가는 습관.

**왜 새는가** thinking 토큰은 생성될 때 output 토큰으로 1회 과금된다. effort를 높게 유지한 채 단순 작업("변수명 바꿔줘")을 시키면 불필요하게 긴 thinking이 매 턴 새로 생성되어 output 토큰을 낭비한다. (최신 모델인 Opus 4.5+/Sonnet 4.6+에서는 이전 턴의 thinking 블록이 컨텍스트에 유지되지만, 이는 캐시 read로 취급되어 풀 input보다 저렴하다. 그 이전 모델에서는 이전 turn의 thinking 블록이 다음 요청 전 제거된다. 어느 쪽이든 "이중 과금"은 아니다.)

**자각신호** 세션 초반 복잡 작업으로 thinking을 크게 쓴 뒤 단순 작업이 이어짐, 단순 요청인데도 응답·비용이 무거움, effort 레벨을 바꾸지 않고 세션 전체 진행.

**고치는습관** 단순 작업으로 전환 시 `/effort low`로 낮추거나 thinking을 끈다(단축키는 환경별 상이 — `/config` 또는 환경변수 `MAX_THINKING_TOKENS=0` 활용). 복잡 설계 작업과 단순 구현 작업은 세션을 분리.

**절감추정** 단순 작업에 불필요하게 큰 thinking을 유발하면 turn당 수천~수만 output 토큰 낭비. effort를 작업에 맞추면 단순 작업 추론 비용 30~60% 절감 가능(정성적).

---

### H4-02 · 과한 Thinking 유도 [R2 D3]
단순 작업에 "깊이 분석해줘", "모든 측면에서 검토해줘" 같은 표현을 쓰는 습관.

**왜 새는가** "깊이 분석", "모든 측면", "완벽한" 같은 표현은 Claude에게 extended thinking 또는 장문 추론 응답을 유도한다. 변수명 변경처럼 단순한 작업에 500~1,000 토큰의 추론을 소모하는 비대칭이 발생한다.

**자각신호** 응답 앞에 장문의 "생각해보면..." 도입부, 단순 결정에 스크롤 3회 이상 필요, "최선의 방법", "완벽한 솔루션" 표현 사용 후 과한 응답.

**고치는습관** 단순 결정은 1줄 응답 요청: "둘 중 더 명확한 걸 골라줘. 이유 1줄만." `/effort low`로 추론 수준 낮추기(thinking 토글 단축키는 환경별 상이 — `/config`로 확인).

**절감추정** 단순 작업의 추론 비용 70~90% 감소.

---

### H4-03 · 캐시 킬 스위치 [R1 P-14]
세션 중 모델을 자주 전환하거나 effort 레벨을 반복 변경하는 습관.

**왜 새는가** 프롬프트 캐싱은 모델명·API 파라미터가 동일할 때만 캐시 히트가 발생한다. 모델 전환 또는 thinking 파라미터 변경은 캐시 브레이크포인트를 무효화해 다음 요청에서 CLAUDE.md + 시스템 툴 스키마(≈21K 토큰) 전체를 풀 비용으로 재과금한다.

**자각신호** 세션 중 모델 또는 effort 레벨 2회 이상 변경, 변경 직후 응답이 느려짐(캐시 미스로 재계산), `/usage`에서 캐시 히트율 갑자기 낮아짐.

**고치는습관** 세션 시작 전 모델과 effort 레벨 결정, 세션 내 변경 최소화. 다른 모델이 필요하면 새 세션을 여는 것이 캐시 비용 측면에서 유리. CLAUDE.md를 자주 편집하면 캐시 무효화 반복 → 편집을 배치로 몰아서 처리.

**절감추정** 모델 전환 1회 시 ≈21,000 토큰 캐시 미스 비용 추가 발생. 세션에서 3회 전환이면 63K 토큰 추가 낭비.

---

## H5 — 프롬프트 명료성 부족 (Prompt Clarity)

---

### H5-01 · 안개 프롬프트 [R2 A1]
"이 코드 좀 개선해줘", "로그인 버그 고쳐줘"처럼 모호하게 요청하는 습관.

**왜 새는가** Claude가 명료화 질문을 하거나 넓은 범위를 탐색하며 수십 개 파일을 읽은 뒤 광범위한 분석을 출력한다. 사용자가 "아 그게 아니라..."로 정정하면 왕복이 발생한다. 공식 문서: "Vague prompts generate expensive sessions."

**자각신호** Claude가 "어떤 파일을 보면 될까요?" 되묻기, 첫 응답이 2~3개 파일에 걸친 장문 분석, 응답 후 "아 그게 아니라"를 타이핑.

**고치는습관** 파일 경로를 `@` 참조로 직접 지정. 증상·위치·완료 기준을 한 번에 명시. 검증 조건(테스트 통과)까지 포함하면 왕복 0회.

**절감추정** 왕복 1~3회 제거 → 프롬프트당 500~2,000 토큰 절감. 광범위 파일 탐색 차단으로 추가 절감.

---

### H5-02 · 무제한 탐색 요청 [R2 A2 / R1 P-12]
"이 코드베이스 분석해줘", "인증 시스템 어떻게 돼 있어?" 처럼 범위 없이 탐색을 시키는 습관.

**왜 새는가** 범위 미지정 탐색은 Claude가 수십~수백 개 파일을 순차 Read한다. 파일 내용이 컨텍스트에 누적되어 이후 구현 작업의 모든 턴 input에 포함된다.

**자각신호** Claude가 "파일 구조를 먼저 파악하겠습니다"라며 ls/find를 수차례 실행, 응답에 "src/X.ts, src/Y.ts..."가 10개 이상 나열, 탐색 후 컨텍스트 사용률 30~40% 점프.

**고치는습관** 탐색 범위를 파일·함수 단위로 명시. "서브에이전트를 써서 X를 탐색해줘" — 별도 컨텍스트에서 탐색 후 요약만 메인으로 반환. 또는 plan mode를 저비용 탐색·계획 대안으로 활용: `Shift+Tab`으로 plan mode 진입 → Claude가 파일을 읽되 수정하지 않으므로 탐색·계획 단계의 output 토큰이 최소화됨 → 계획 확정 후 일반 모드로 전환해 구현. 공식 권고: 작고 명확한 작업은 plan 생략, 멀티파일 변경은 plan mode 우선.

**절감추정** 광범위 탐색 1회 ≈ 10,000~50,000 토큰. 서브에이전트 격리 시 70~95% 절감.

---

### H5-03 · 분할 요구 결핍 [R2 A3]
요구사항을 조금씩 나눠 여러 번 추가하는 습관. "아 소셜 로그인도 넣어야 해", "폼 검증도 추가해줘" 식의 연속 추가.

**왜 새는가** 각 후속 요청마다 이전 대화 전체(점점 늘어나는 컨텍스트)를 재처리하며 응답한다. 4회 왕복 시 1회차 대비 컨텍스트 누적량이 3~5배.

**자각신호** 하나의 기능 구현에 동일 세션 내 4회 이상 추가 요청, "아 그리고...", "하나만 더..." 패턴 반복.

**고치는습관** 작업 시작 전 요구사항을 일괄 명세. 공식 인터뷰 패턴: "로그인 페이지 만들기 전에 필요한 기능을 다 물어보고 SPEC.md를 써줘." SPEC.md를 새 세션에서 소비하면 히스토리 오염 0.

**절감추정** 4회 왕복 → 1회 단축 시 컨텍스트 누적 70~80% 감소.

---

### H5-04 · 장황 출력 유도 [R2 A4]
"이 함수 설명해줘"처럼 출력 범위를 제한하지 않고 요청하는 습관.

**왜 새는가** 열린 요청은 Claude가 배경·원리·예시·경고·대안을 모두 포함한 장문을 출력하게 만든다. output 토큰은 input보다 비싸고, 이 출력이 다시 다음 턴의 컨텍스트로 누적된다. 공식 문서: "The difference can represent 40-60% of token usage."

**자각신호** 응답이 스크롤을 내려야 할 만큼 길지만 첫 단락만 읽음, "요약만 해줘"를 사후에 요청(이미 생성된 후), 응답에 "참고로...", "추가적으로..." 단락 3개 이상.

**고치는습관** "이 함수 목적을 2줄로만. 코드·예시 없이." CLAUDE.md에 출력 기본값 설정(전체 세션에 영속 적용): "코드 설명: 3줄 이하. 불필요한 서론·배경 생략." tool(파일 읽기·Bash 등)이 필요 없는 순수 질의는 `/btw` 활용 — 답변이 컨텍스트에 추가되지 않는다(단, `/btw`는 tool을 쓰지 못하므로 이미 알려진 정보에 대한 질의 전용).

**절감추정** 응답 토큰 40~60% 감소. CLAUDE.md 설정 1회로 모든 세션에 영속 적용.

---

## H6 — 재작업 루프 (Rework Loop)

---

### H6-01 · 전체 재생성 요청 [R2 B1]
부분 수정이면 될 걸 "다시 짜줘"로 전체 재출력을 유발하는 습관.

**왜 새는가** Claude는 기존 코드를 다시 읽고 전체를 재출력한다. 200줄 파일이면 200줄 재생성. 실제 변경은 10줄인데도. 재출력된 코드가 다시 컨텍스트에 쌓인다. 공식: "Re-requesting reformatted content doubles the token cost."

**자각신호** 응답이 전체 파일·함수를 코드블록으로 다시 출력, 변경점이 실제로 5% 미만인데 100% 재출력, 직전 응답과 새 응답 diff 해보면 수정 부분이 극히 적음.

**고치는습관** "위 코드에서 35~42줄에만 try/catch 추가해줘. 나머지는 그대로." 또는 "@src/api.ts 파일의 fetchUser 함수에만 에러 처리 추가. 파일 전체 재출력 하지 마." 코드 리뷰 시 "diff만 보여줘" 명시.

**절감추정** 100줄 파일 기준 90% 출력 토큰 절감. 파일 재독 비용 제거.

---

### H6-02 · 반복 검증 요청 [R2 B2]
구현 후 "정말 맞아?", "한번 더 확인해줘", "혹시 엣지케이스 빠진 거 없어?" 를 연속으로 하는 습관.

**왜 새는가** Claude는 각 검증 요청마다 자신이 작성한 코드를 다시 읽고 분석한다. 3회 반복이면 동일 코드를 3번 추가 처리. Claude는 자신이 구현한 코드를 검증할 때 확증 편향이 있어 검증 효과도 낮다.

**자각신호** "정말?", "확실해?", "다시 봐줘" 패턴이 구현 후 연속 발생, Claude가 "네, 맞습니다" → "아 잠깐..." 패턴 반복.

**고치는습관** 구현 요청 시 검증 조건을 미리 포함: "구현 후 기존 테스트 실행하고 결과 보여줘." 또는 서브에이전트 코드 리뷰: "서브에이전트로 방금 작성한 코드의 엣지케이스를 검토해줘. 정확성에 영향 있는 것만 보고해." `/code-review` 슬래시 커맨드 사용.

**절감추정** 반복 검증 3회 → 1회(자동화된 검증)로 대체. 왕복 2회 × 코드 재독 비용 제거.

---

## H7 — 영속화 미활용 (Persistence Underuse)

---

### H7-01 · 맥락 재타이핑 [R2 C1]
매 세션 시작 시 프로젝트 스택·규칙·컨벤션을 매번 손으로 입력하는 습관.

**왜 새는가** CLAUDE.md를 활용하지 않으면 매 세션마다 동일 맥락을 타이핑하게 되고, 이 반복 입력이 컨텍스트 앞부분을 차지해 실제 작업 공간을 줄인다. 세션마다 조금씩 달라져 일관성도 낮아진다.

**자각신호** 세션 첫 번째 메시지가 항상 프로젝트 설명으로 시작, "아까도 말했지만...", "이전에 얘기한 것처럼..." 표현, 동일 스택·규칙 설명 3회 이상 반복.

**고치는습관** `/init` 실행 → 생성된 CLAUDE.md에 반복 입력하던 내용 추가. 반복 작업 패턴은 skill(`.claude/skills/`)로 추가 영속화.

**절감추정** 세션당 맥락 재타이핑 100~300 토큰 × 일 3~5 세션 = 주당 1,500~7,500 토큰 완전 제거.

---

### H7-02 · 커스텀 명령 미활용 [R2 C2]
반복 워크플로우(PR 생성·배포 체크·코드 리뷰 절차 등)를 매번 장문으로 입력하는 습관.

**왜 새는가** 반복 워크플로우를 매번 장문으로 타이핑하면 입력 토큰 낭비. 지시가 매번 조금씩 달라 일관성 저하. 빠진 단계가 생겨 재요청 발생.

**자각신호** 같은 작업 절차를 세션마다 장문으로 타이핑, "우리 컨벤션은 이렇게..." 설명을 매번 붙임, 단계 하나를 빠뜨려 "아 그리고 린트도..." 추가 요청.

**고치는습관** `.claude/skills/create-pr/SKILL.md` 생성 → 이후 `/create-pr` 한 줄로 완료. `/init`으로 시작, 반복 패턴 발견 시 즉시 skill로 추출하는 습관.

**절감추정** 반복 워크플로우 장문(100~300 토큰) → 1줄 명령어. 일관성 향상으로 재요청 방지.

---

### H7-03 · 출력 휘발 [R2 C3]
분석·계획·스펙 등 중간 산출물을 파일로 저장하지 않고 대화창에만 두는 습관.

**왜 새는가** 같은 세션 내 재참조 시 컨텍스트에 이미 있는 내용을 다시 생성하는 낭비. 새 세션에서 다시 생성해야 하는 비용. 공식 권고: "Once the spec is complete, start a fresh session to execute it."

**자각신호** Claude의 분석/계획 결과가 대화창에만 있고 파일 없음, "아까 말한 대로", "방금 분석한 것처럼"으로 이전 출력 재참조, 새 세션에서 "저번에..." → 결과물 없어서 재작업.

**고치는습관** "인증 시스템 분석해서 ANALYSIS.md로 저장해줘." "스펙 완성되면 SPEC.md 저장하고, 새 세션에서 이걸 기반으로 구현할게." 모든 중간 산출물(분석·계획·스펙·결정)은 파일로 저장. 새 세션은 파일을 `@` 참조로 소비.

**절감추정** 중간 산출물 재생성 방지. 세션 간 산출물 공유로 동일 작업 반복 제거.

## H8 — 컨텍스트 오염 (Context Pollution)

---

### H8-01 · 메인 스레드 탐색 [R2 D1 / R1 P-09]
구현 중인 메인 세션에서 "기존 OAuth 처리 어떻게 했는지 찾아봐줘" 같은 탐색을 시키는 습관.

**왜 새는가** 메인 세션에서 파일 탐색을 하면 탐색한 모든 파일 내용이 메인 컨텍스트에 쌓인다. 탐색 목적이 달성된 후에도 그 파일들은 컨텍스트에 남아 이후 모든 턴의 처리 비용을 높인다. 서브에이전트는 별도 컨텍스트 창에서 실행되고 요약만 메인으로 반환.

**자각신호** 메인 세션에서 "파일 구조 파악하겠습니다"가 구현 중간에 등장, 컨텍스트 사용률 표시가 탐색 후 급증, 탐색용 파일 내용이 이후 응답에도 재참조됨.

**고치는습관** 새로 파일을 찾아야 하는 탐색은 "서브에이전트로 src/auth/ 디렉토리에서 OAuth 관련 함수 찾아서 함수명·위치만 요약해줘." 이미 컨텍스트에 있는 정보에 대한 단순 질의는 `/btw` 사용(tool을 쓰지 못하고 기존 컨텍스트만 참조하므로 새 파일 탐색에는 부적합, 히스토리에 추가 안 됨). `.claude/agents/`에 전용 조사 서브에이전트 정의(tools: Read, Grep, Glob만 허용).

**절감추정** 탐색 파일들이 메인 컨텍스트에서 제거. 탐색 35,000~50,000 토큰 vs 서브에이전트 요약 3,000 토큰 반환 → 90%+ 절감.

---

### H8-02 · stdout 홍수 [R1 P-13]
빌드·테스트 전체 출력을 필터 없이 Claude에게 실행시키는 습관.

**왜 새는가** Bash 툴이 실행한 명령의 stdout 전체가 툴 결과로 컨텍스트에 영구 추가된다. 10,000줄 테스트 로그 ≈ 수만 토큰이 이후 모든 턴의 input으로 재과금된다.

**자각신호** 빌드/테스트 실행 후 컨텍스트 사용률 급등, 테스트 통과 케이스 수백 개가 로그에 포함, Claude가 "빌드 성공" 확인을 위해 전체 로그를 읽어야 함.

**고치는습관** PreToolUse 훅 설정: `grep -A5 'FAIL\|ERROR\|error:'`로 실패만 필터(공식 docs filter-test-output.sh 예시). 프롬프트에 출력 제한 명시: "테스트 실행 후 실패한 케이스만 보고해줘." 서브에이전트로 테스트 실행 위임.

**절감추정** 10,000줄 로그 → grep 결과 30줄 = 99%+ 절감. 이후 10턴 재과금 방지까지 세션당 수십만 토큰.
