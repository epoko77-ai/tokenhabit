# 수동(Passive) 컨텍스트 누적 토큰 낭비 패턴 연구

**담당**: R1 — 수동 컨텍스트 누적 계열  
**기준일**: 2026-06-03  
**검증 소스**: docs.anthropic.com, code.claude.com/docs, claudefa.st, damiangalarza.com, jdhodges.com, dev.to 등

---

## 카테고리 A — 세션 위생 (Session Hygiene)

작업이 끝났거나 맥락이 달라진 상황에서 컨텍스트를 제때 정리하지 않아 발생하는 낭비.

---

### P-01 · 무관 작업 연속 진행 ("주제 드래그")

**무의식 시나리오**  
로그인 버그를 고친 뒤 `/clear` 없이 바로 "이제 대시보드 CSS 좀 고쳐줘"를 입력한다. 두 작업은 파일·목적·도메인이 전혀 다르다.

**왜 토큰이 새는가**  
Claude Code는 매 턴마다 전체 대화 히스토리를 API에 재전송한다(stateless 구조). 새 CSS 작업을 시작하는 순간부터 이전 버그픽스의 모든 파일 읽기 결과·에러 스택·대화 내용이 매 턴 input 토큰으로 과금된다. 공식 문서: "Claude Code's context window holds your entire conversation, including every message, every file Claude reads, and every command output."

**자각 신호(tell)**  
- `/usage`에서 Messages 비율이 40% 이상  
- Claude가 현재 작업과 무관한 파일을 "이전에 읽었다"며 언급함  
- 세션 지속 시간이 60분 이상인데 다른 주제가 섞여 있음

**고치는 습관(fix)**  
- 작업 전환 전에 항상 `/clear` 실행 (컨텍스트 윈도우 완전 초기화)  
- `/rename`으로 현재 세션에 이름 붙인 뒤 `/clear` → 나중에 `/resume`으로 복귀 가능  
- 공식 권고: "Use `/clear` frequently between tasks to reset the context window entirely" (code.claude.com/docs/en/best-practices)

**절감 추정**  
작업 전환 후 10~20턴 진행 시, 이전 컨텍스트 크기만큼 매 턴 절감. 평균 50K 토큰짜리 선행 세션이 붙어있으면 후속 작업 전체에서 50K × 턴수만큼 중복 과금. 정성적으로 **2~5× 비용 절감** 가능.

---

### P-02 · 오류 루프 후 /compact 미실행 ("실패 컨텍스트 적체")

**무의식 시나리오**  
Claude가 같은 오류를 두 번 다르게 잘못 고쳤다. 사용자는 계속 "아직도 안 돼", "이번에도 틀렸어"를 입력한다. 히스토리가 실패한 시도들의 코드·에러 메시지·설명으로 점점 두꺼워진다.

**왜 토큰이 새는가**  
모든 실패 시도(코드 조각, 에러 텍스트, 재시도 설명)가 히스토리에 쌓이고 매 턴 재전송된다. 실패한 접근법은 이후 추론에 도움이 되기는커녕 "lost in the middle" 효과로 정확도를 떨어뜨린다. 두 번 연속 보정 후에도 문제가 지속되면 공식 문서는 명시적으로 `/clear` 권고.

**자각 신호(tell)**  
- 같은 이슈로 3턴 이상 보정 중  
- Claude가 이미 시도해 실패한 방법을 다시 제안함  
- `/usage` 메시지 토큰이 빠르게 증가

**고치는 습관(fix)**  
- 2번 연속 보정 실패 시 즉시 `/compact "실패한 접근법 제외, 현재 코드 상태와 문제만 보존"`  
- 또는 `/clear` 후 더 구체적인 초기 프롬프트로 재시작  
- `/rewind` (Esc+Esc)로 실패 이전 체크포인트로 롤백

**절감 추정**  
실패 루프 5턴 × 평균 20K 누적 히스토리 = 100K 추가 input 토큰. `/compact` 타이밍에 따라 **40~70% 절감** 가능.

---

### P-03 · 자동 compaction 의존 ("compaction 버스 막차 타기")

**무의식 시나리오**  
세션을 오래 끌다가 화면 하단에 "Auto-compacting context..." 메시지가 뜬 뒤에야 인식한다. 그 전 수십 턴은 200K 가까운 풀 컨텍스트를 재전송한 상태다.

**왜 토큰이 새는가**  
자동 compaction은 83.5% 임계점(200K 윈도우 기준 약 167K 토큰)에서 발동한다. 그 이전까지는 전체 히스토리가 매 턴 input으로 과금된다. 컨텍스트가 100K → 167K 구간에서 진행한 모든 턴이 비례적으로 비싸진다.

**자각 신호(tell)**  
- `/usage` 또는 상태바에서 컨텍스트 사용률 50% 초과  
- 세션 지속이 60분 이상이고 복잡한 주제가 오래 이어짐  
- 응답 속도가 느려지고 Claude가 초반 지시를 "잊는" 현상

**고치는 습관(fix)**  
- 컨텍스트 50~60% 시점에 `/compact [포커스 지시]` 수동 실행  
- 예: `/compact Focus on API changes and modified file list`  
- 상태바 토큰 표시 설정 (`/statusline` 또는 커스텀 statusline 설정으로 실시간 모니터링)  
- CLAUDE.md에 compaction 지시 추가: "When compacting, always preserve the full list of modified files and test commands"

**절감 추정**  
100K → 167K 구간의 30턴이라면, 평균 133K input/턴 × 30 = 4M 추가 토큰. 50%에 미리 compact하면 그 이후 turnback 비용을 **절반 이하**로 유지.

## 카테고리 B — 반복 입력 낭비 (Redundant Input)

이미 컨텍스트에 있거나 파악 가능한 정보를 재주입해 토큰이 새는 패턴.

---

### P-04 · 같은 파일 반복 읽기 ("파일 리드 재탕")

**무의식 시나리오**  
"auth.ts 다시 보여줘", "그 파일 한 번 더 확인해", "아까 그 파일에서 토큰 리프레시 로직"처럼, 한 세션에서 같은 파일을 여러 번 Read 요청한다.

**왜 토큰이 새는가**  
Claude Code의 Read 툴은 파일 전체를 툴 결과로 컨텍스트에 추가한다. 200줄 파일(≈2,000 토큰)을 5번 읽으면 동일 내용이 10,000 토큰으로 부풀어 매 후속 턴에 재전송된다. 파일 내용이 이미 컨텍스트에 있어도 Claude Code는 "다시 읽어라"는 지시에 별도 툴 호출을 발생시킨다.

**자각 신호(tell)**  
- 같은 파일명이 툴 호출 로그에 반복 등장  
- Read 결과가 대화에 여러 번 삽입됨  
- `/usage`에서 툴 결과(tool output)가 메시지 비율을 압도

**고치는 습관(fix)**  
- "아까 읽은 auth.ts에서 토큰 리프레시 부분 보여줘"처럼 파일 재읽기 없이 컨텍스트 내 참조 유도  
- `read-once` 훅 도입: PreToolUse 훅으로 이미 읽은 파일 재읽기를 차단하고 변경분만 diff로 반환 (80~95% 토큰 절감 보고됨)  
- 코드 인텔리전스 플러그인 사용: "go to definition" 한 번으로 grep + 다수 파일 읽기를 대체

**절감 추정**  
200줄 파일 5회 재읽기 = 약 8,000 토큰 절약. 대형 파일(500줄+)에서 반복 빈번하면 **세션당 수만 토큰** 절감.

---

### P-05 · 대형 출력 통째 붙여넣기 ("로그 전체 덤프")

**무의식 시나리오**  
빌드 실패 후 터미널 전체 출력(10,000줄 빌드 로그)이나 긴 스택트레이스를 Ctrl+A → Ctrl+V로 Claude 프롬프트에 붙여넣는다. 실제로 필요한 건 마지막 20~30줄의 에러 메시지다.

**왜 토큰이 새는가**  
붙여넣기한 텍스트는 사용자 메시지로 히스토리에 영구 추가된다. 10,000줄 로그 ≈ 수만 토큰이 이후 모든 턴의 input에 포함된다. 또한 Claude Code의 Bash 툴이 실행한 명령의 전체 stdout도 툴 결과로 컨텍스트에 쌓인다: "A 10,000-line log stays in context for every subsequent message."

**자각 신호(tell)**  
- 프롬프트 입력 시 수천 줄 텍스트를 그대로 붙임  
- Bash 툴로 실행한 테스트/빌드 출력이 필터링 없이 노출됨  
- 다음 턴부터 응답이 현저히 느려짐

**고치는 습관(fix)**  
- 붙여넣기 전: 에러 메시지 핵심 20~30줄만 수동 선별  
- 툴 출력 필터링: PreToolUse 훅으로 테스트 실패 시 `grep -A5 'FAIL\|ERROR'`만 반환하도록 사전 필터링 (공식 docs 예시 제공)  
- 파일로 먼저 저장 후 경로만 전달: "에러 로그 `/tmp/build.log`에 저장했어. 분석해줘"  
- `cat error.log | claude` 파이프 방식으로 Claude Code에 직접 스트림 전달 (단, 필터 적용 필수)

**절감 추정**  
10,000줄 로그 → 30줄로 필터 = **99% 이상 절감**. 해당 내용이 이후 10턴에 재과금되는 걸 감안하면 수십만 토큰 차이.

---

### P-06 · 이미지/스크린샷 반복 첨부 ("스크린샷 중독")

**무의식 시나리오**  
UI 버그를 설명할 때마다 전체 화면 스크린샷을 첨부한다. "이거 고쳐줘 [스크린샷]", "아직 이상해 [스크린샷]", "버튼이 이렇게 보여 [스크린샷]"처럼 반복한다.

**왜 토큰이 새는가**  
이미지는 픽셀 수에 비례한 토큰을 소비한다. 1,000×1,000px 스크린샷 ≈ 1,334 토큰. 이미지는 히스토리에 남아 이후 턴에 재전송된다. Opus 계열에서는 동일 이미지가 3× 더 비싸다. 텍스트로 "버튼이 오른쪽으로 2px 밀렸음"을 표현할 수 있는 정보에 수천 토큰을 쓰는 셈이다.

**자각 신호(tell)**  
- 하나의 이슈 해결에 3개 이상 이미지를 첨부  
- 이미지 내 관심 영역이 전체 화면의 10% 미만  
- 에러 메시지가 텍스트인데 스크린샷으로 첨부

**고치는 습관(fix)**  
- 텍스트로 대체 가능하면 텍스트로: 에러 메시지는 복사+붙여넣기  
- 이미지 필수 시 크롭 후 첨부: 200×200px ≈ 54 토큰 (1,000×1,000 대비 25배 절감)  
- 동일 스크린샷을 재첨부하지 않기: "아까 첨부한 스크린샷 기준으로..."로 컨텍스트 내 참조

**절감 추정**  
1,000×1,000px 스크린샷 5회 반복 ≈ 6,670 토큰. 크롭 + 텍스트 대체 시 **90% 이상 절감**. Opus 요금 기준으로 세션당 수달러 차이 가능.

## 카테고리 C — 상시 주입 컨텍스트 (Always-On Context Injection)

세션 시작부터 매 턴 자동 주입되는 컨텍스트가 불필요하게 큰 패턴.

---

### P-07 · 과비대 CLAUDE.md ("설명서가 곧 예산")

**무의식 시나리오**  
프로젝트 규칙, 아키텍처 설명, API 문서, 도메인 지식, 팀 컨벤션을 CLAUDE.md 하나에 몽땅 몰아넣는다. 파일이 260줄, 4,000+ 토큰에 달한다.

**왜 토큰이 새는가**  
CLAUDE.md는 세션 시작 시 자동 로드되어 매 턴 input의 고정 비용이 된다. 프롬프트 캐싱 덕분에 첫 요청 이후 캐시 히트 시 10%로 과금되지만, 5분 캐시 TTL(기본) 초과 후 재과금된다. 불필요한 내용이 길어질수록 Claude가 실제로 필요한 지시를 "잃어버리는" (lost in the middle) 현상도 심화. 공식 권고: "Keep it under 200 lines."

**자각 신호(tell)**  
- CLAUDE.md가 200줄 초과  
- Claude가 CLAUDE.md에 명시한 규칙을 무시하는 빈도 증가  
- `/usage`에서 Memory files 비율이 3% 초과  
- 현재 작업과 무관한 도메인 섹션이 항상 포함됨

**고치는 습관(fix)**  
- CLAUDE.md는 200줄·2,000 토큰 미만으로 유지  
- 특정 작업에만 필요한 지식은 Skills로 분리 (`.claude/skills/`): 온디맨드 로드, 미사용 시 이름/설명만 ≈200 토큰  
- `@path` 임포트는 필요 시점에 명시적으로만: 자동 임포트는 해당 파일 전체를 항상 로드  
- 캐시 무효화 방지: CLAUDE.md를 자주 수정하면 캐시 미스 → 매 세션 풀 비용 재과금

**절감 추정**  
4,000 토큰짜리 CLAUDE.md → 1,000 토큰으로 감량 = 세션당 3,000 토큰 고정 절감. 10턴 세션이면 3만 토큰, 100턴이면 30만 토큰 절감.

---

### P-08 · 안 쓰는 MCP 서버 상시 연결 ("툴 스키마 정기권")

**무의식 시나리오**  
Playwright, Gmail, Slack, Notion, SQLite 등 MCP 서버를 한꺼번에 연결해놓고, 실제 세션에서는 그 중 일부만 사용한다. "혹시 필요할지 몰라서" 전부 켜둔다.

**왜 토큰이 새는가**  
MCP 툴 정의(schema)는 매 API 요청마다 컨텍스트에 주입된다. Claude Code의 tool search(defer)가 기본 활성화되어 있지만, 서버가 연결된 이상 툴 이름 목록은 매 턴 전송된다. 실측: Playwright ≈3,442 토큰, Gmail ≈2,640 토큰. 5개 서버 조합 시 세션 시작 전 55,000 토큰이 소비된 사례도 보고됨. 공식 문서: "Disable unused servers: Run `/mcp` to see configured servers and disable any you're not actively using."

**자각 신호(tell)**  
- `/context`에서 System tools 비율 10% 초과  
- 세션에서 실제로 쓰지 않는 MCP 툴이 절반 이상  
- 단순 코딩 세션인데도 대화 시작이 느림

**고치는 습관(fix)**  
- `/mcp` 명령으로 현재 연결 서버 확인, 미사용 서버 비활성화  
- 특정 작업 전용 MCP는 해당 세션에만 활성화  
- CLI 대체 가능한 경우 MCP보다 CLI 툴 우선 (gh, aws, gcloud는 schema 주입 없음)  
- Tool search(defer)는 이미 기본 활성화 상태이므로 서버 수 자체를 최소화하는 것이 핵심

**절감 추정**  
4서버 연결 시 ≈7,077 토큰/턴 고정 오버헤드. 2서버만 유지하면 ≈3,000~4,000 토큰/턴 절감. 100턴 세션 기준 **30~40만 토큰** 차이.

### P-09 · 서브에이전트에 무거운 탐색 맡기지 않기 ("메인 컨텍스트 탐사대")

**무의식 시나리오**  
"인증 모듈이 어떻게 돌아가는지 파악해봐"처럼 광범위한 탐색 작업을 메인 세션에서 직접 시킨다. Claude가 수십 개 파일을 읽고 그 내용이 모두 메인 컨텍스트에 쌓인다.

**왜 토큰이 새는가**  
메인 세션에서 파일을 읽으면 그 내용 전체가 메인 컨텍스트의 히스토리로 영구 추가되어 이후 모든 턴에 재과금된다. 서브에이전트는 별도 컨텍스트 윈도우에서 실행되고, 완료 후 요약(2,000~4,000 토큰)만 메인 세션에 반환한다. 탐색 비용(35,000~50,000 토큰)이 서브에이전트 격리 컨텍스트 안에서 소멸된다.

**자각 신호(tell)**  
- 탐색/리서치 요청 이후 메인 컨텍스트가 급격히 증가  
- "X가 어떻게 작동하는지 파악해줘" 유형의 열린 탐색 요청  
- 구현 전 조사가 메인 세션에서 이루어짐

**고치는 습관(fix)**  
- "Use subagents to investigate X"로 탐색을 격리: 서브에이전트가 파일 읽고 요약만 반환  
- `.claude/agents/` 에 전용 조사 서브에이전트 정의 (tools: Read, Grep, Glob만 허용)  
- plan mode(Shift+Tab)로 읽기만 하는 탐색 단계 분리  
- `/btw` 명령으로 단발성 질문 처리: 답변이 히스토리에 남지 않음

**절감 추정**  
탐색 시 파일 읽기 35,000 토큰 vs 서브에이전트 요약 3,000 토큰 반환. **90% 이상 절감**. 메인 컨텍스트 오염 방지 효과까지 고려하면 후속 턴 절감도 복리로 누적.

---

## 카테고리 D — thinking 토큰 누적 (Thinking Token Accumulation)

추론 토큰이 히스토리에 쌓여 재과금되는 패턴.

---

### P-10 · thinking 블록 세션 내 재과금 ("추론의 망령")

**무의식 시나리오**  
Sonnet 4.6 또는 Opus 4.5+ 모델에서 effort=high(기본값)로 작업한다. Claude가 복잡한 코드 설계를 추론하며 수천 토큰의 thinking 블록을 생성했고, 그 뒤 간단한 "변수명 바꿔줘" 같은 단순 작업도 같은 세션에서 이어서 한다.

**왜 토큰이 새는가**  
Sonnet 4.6과 Opus 4.5+ 모델에서는 이전 턴의 thinking 블록이 컨텍스트에 보존되어 다음 턴 input 토큰으로 재과금된다. API 문서: "Thinking blocks from prior assistant turns are kept in context on Opus 4.5+ and Sonnet 4.6+ (as input tokens when read from cache)." thinking 블록은 output 토큰으로 최초 과금되고, 이후 캐시 input 토큰으로 재과금되는 이중 비용 구조다. 단, thinking 파라미터 변경 시 캐시 무효화로 풀 비용 재발생.

**자각 신호(tell)**  
- 세션 초반 복잡한 작업으로 thinking이 많이 발생한 뒤 단순 작업이 이어짐  
- 단순 요청인데도 API 비용이 생각보다 높음  
- effort 레벨을 바꾸지 않고 세션 전체를 진행

**고치는 습관(fix)**  
- 단순 작업으로 전환 시 `/effort low` 또는 `/effort medium`으로 낮춤  
- 또는 `MAX_THINKING_TOKENS=0` 환경변수로 thinking 완전 비활성화 (Option+T 토글)  
- 복잡 설계 작업과 단순 구현 작업은 세션을 분리해 thinking 블록 누적 차단  
- `/config`에서 thinking mode 글로벌 기본값 설정

**절감 추정**  
thinking 블록 1턴 ≈ 5,000~20,000 output 토큰 + 이후 매 턴 input 재과금. 10턴 세션 기준 thinking 비활성화 시 **30~60% 비용 절감** 가능.

## 카테고리 E — 구조적 컨텍스트 팽창 (Structural Context Bloat)

사용자 행동보다 프로젝트 설정·습관에서 발생하는 만성 낭비.

---

### P-11 · 거대 reference 파일 매번 전체 주입 ("문서 카탈로그 구독")

**무의식 시나리오**  
CLAUDE.md에 `@docs/full-api-spec.md`, `@architecture/overview.md` 같은 대형 파일을 임포트해 놓는다. 한 세션에서 API 관련 작업을 하나만 해도 전체 API 명세서가 매 턴 로드된다.

**왜 토큰이 새는가**  
CLAUDE.md의 `@path` 임포트는 조건부 로드가 아닌 자동 전체 로드다. 해당 파일의 모든 내용이 세션 시작 시 컨텍스트에 추가된다. 필요한 건 API 명세서의 로그인 엔드포인트 한 섹션뿐인데 전체 명세(5,000+ 토큰)가 매 턴 상주한다.

**자각 신호(tell)**  
- CLAUDE.md에 `@` 임포트 항목이 3개 이상  
- 임포트된 파일 중 현재 작업과 무관한 것이 있음  
- `/context`에서 Memory files 비율이 높음

**고치는 습관(fix)**  
- CLAUDE.md에서 `@` 임포트 제거, 필요 시 프롬프트에서 직접 `@docs/api-spec.md#login` 참조  
- 대형 레퍼런스는 Skill로 전환: "api-conventions" 스킬에 핵심만 발췌, 온디맨드 호출  
- 작업 시작 시 해당 작업에 필요한 파일만 명시적으로 지정  
- Claude를 프로젝트 서브디렉토리에서 실행해 자동 탐색 범위 제한

**절감 추정**  
5,000 토큰짜리 참조파일 2개 제거 = 세션당 10,000 토큰 고정 절감. **정성적으로 가장 간단한 개선**이면서 매 세션 복리 효과.

---

### P-12 · 광범위 탐색 프롬프트 ("무한 그레이징")

**무의식 시나리오**  
"이 코드베이스 좀 파악해줘", "인증 시스템이 어떻게 되는지 전반적으로 설명해줘"처럼 범위를 정하지 않은 탐색을 메인 세션에 시킨다. Claude가 수십 개 파일을 자율적으로 읽는다.

**왜 토큰이 새는가**  
범위 없는 탐색은 Claude가 휴리스틱으로 관련될 수 있다고 판단하는 파일을 모두 읽게 만든다. 모든 파일 읽기 결과가 툴 출력으로 메인 컨텍스트에 누적되어 이후 구현 작업의 모든 턴 input에 포함된다. "파악"하는 데 든 수만 토큰이 실제 구현과 무관함에도 영구 부담이 된다.

**자각 신호(tell)**  
- 프롬프트에 "전반적으로", "전체적으로", "파악해줘" 같은 열린 표현  
- 탐색 후 컨텍스트 사용률이 갑자기 30~40% 점프  
- Claude가 불필요해 보이는 파일까지 읽어들임

**고치는 습관(fix)**  
- 탐색을 서브에이전트로 위임: "Use a subagent to investigate how authentication handles token refresh"  
- 탐색 범위를 파일/함수 단위로 명시: "src/auth/token.ts의 refresh 로직만"  
- plan mode에서 탐색 후 구현 세션 분리  
- Code intelligence 플러그인으로 탐색 비용 자체를 줄임 (grep 대신 symbol navigation)

**절감 추정**  
광범위 탐색 1회 ≈ 10,000~50,000 토큰. 서브에이전트로 격리 시 요약 3,000 토큰만 반환 → **70~95% 절감**.

---

### P-13 · 긴 빌드/테스트 출력 필터 없이 툴 실행 ("stdout 홍수")

**무의식 시나리오**  
`npm test`, `cargo build`, `pytest -v` 등 전체 출력을 그대로 실행하도록 Claude에게 시킨다. 수천 줄의 통과 테스트 결과, 진행 상태, 경고 메시지가 툴 결과로 컨텍스트에 추가된다.

**왜 토큰이 새는가**  
Bash 툴이 실행한 명령의 stdout 전체가 툴 결과로 컨텍스트에 영구 추가된다. 10,000줄 테스트 로그 ≈ 수만 토큰이 이후 모든 턴의 input으로 재과금된다. 공식 문서 예시: PreToolUse 훅으로 테스트 실패만 grep하면 수만 토큰 → 수백 토큰으로 압축.

**자각 신호(tell)**  
- 빌드/테스트 실행 후 컨텍스트 사용률 급등  
- 테스트 통과 케이스 수백 개가 로그에 포함  
- Claude가 "빌드 성공" 확인을 위해 전체 로그를 읽어야 함

**고치는 습관(fix)**  
- PreToolUse 훅 설정: `grep -A5 'FAIL\|ERROR\|error:'`로 실패만 필터 (공식 docs filter-test-output.sh 예시 참조)  
- 프롬프트에 출력 제한 명시: "테스트 실행 후 실패한 케이스만 보고해줘"  
- 성공/실패 exit code만 확인: `npm test > /dev/null && echo OK || echo FAIL`  
- 서브에이전트로 테스트 실행 위임해 출력이 메인 컨텍스트에 오지 않게 격리

**절감 추정**  
10,000줄 테스트 로그 → grep 결과 30줄 = **99% 이상 절감**. 이후 10턴 재과금 방지까지 포함하면 세션당 수십만 토큰.

---

### P-14 · 모델 전환 시 캐시 무효화 무시 ("캐시 킬 스위치")

**무의식 시나리오**  
세션 중 `/model opus`로 전환했다가 다시 `/model sonnet`으로 돌아온다. 또는 `/effort` 레벨을 자주 바꾼다. 이전에 캐시된 시스템 프롬프트·CLAUDE.md·툴 스키마가 무효화되어 풀 비용으로 재과금된다.

**왜 토큰이 새는가**  
프롬프트 캐싱은 모델명·API 파라미터(thinking budget 포함)가 동일할 때만 캐시 히트가 발생한다. 모델 전환 또는 thinking 파라미터 변경은 캐시 브레이크포인트를 무효화해 다음 요청에서 CLAUDE.md + 시스템 툴 스키마(≈21K 토큰) 전체를 다시 풀 비용으로 과금한다. 공식 API 문서: "Changing thinking parameters invalidates message cache breakpoints."

**자각 신호(tell)**  
- 세션 중 모델 또는 effort 레벨을 2회 이상 변경  
- 변경 직후 응답이 느려짐 (캐시 미스로 재계산)  
- `/usage`에서 캐시 히트율이 갑자기 낮아짐

**고치는 습관(fix)**  
- 세션 시작 전 모델과 effort 레벨 결정, 세션 내 변경 최소화  
- `opusplan` 모델 별칭 활용: plan 단계는 opus, 구현 단계는 sonnet을 자동 전환 (캐시 관리 Claude Code가 처리)  
- 다른 모델이 필요하면 새 세션을 여는 것이 캐시 비용 측면에서 유리  
- CLAUDE.md를 자주 편집하면 캐시 무효화 반복 → 편집 배치로 몰아서 처리

**절감 추정**  
모델 전환 1회 시 ≈21,000 토큰 캐시 미스 비용 추가 발생(시스템 툴 스키마 기준). 세션에서 3회 전환이면 **63K 토큰 추가 낭비**.
